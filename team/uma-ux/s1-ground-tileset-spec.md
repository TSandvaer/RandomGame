**Owner:** Uma (vision/direction + gen plan) → orch (PixelLab `create_topdown_tileset` gen, orch-only) → Drew (Godot TerrainSet wiring + autotile painting/placement) · **Phase:** M3 S1 GROUND layer.

The COMPOSITION INTENT of v2 (fine-cobble lane through a varied dirt/grass/cobble yard) **carries forward unchanged**; only the gen method flips from procedural-Voronoi to **AI-gen Wang tilesets + Godot autotile**.

## 0. WHY THIS RESPEC — procedural is the wrong tool (read first)

The procedurally-composed ground has been soak-rejected **~7 times**. Sponsor's binding call this round:

> *Procedurally-composed ground "looks like crap." It is the WRONG TOOL — procedural blob/dither generation can't reach the hand-authored quality of the references. AI-gen a proper ground TILESET with TRANSITION TILES + Godot autotile for seamless blends. This matches the AI-gen pipeline that already produces good characters/mobs.*

**The hard pivot:**
1. **STOP extending `gen_s1_cobble_floor.py` / any procedural ground generator.** The Voronoi-cobble tech is retired as the *ground* solution. (It may survive as a salvageable cobble *texture* source if a Wang base-tile needs warm fine-cobble craft — see §3 — but it is no longer how the ground is composed.)
2. **The ground is an AI-generated Godot TERRAIN SET.** Terrains = DIRT (base field) / GRASS / fine-COBBLE path. Transitions between them are generated by PixelLab **`create_topdown_tileset`** (a Wang/corner-autotile generator built exactly for seamless terrain boundaries), then wired into a Godot `TileSet` TerrainSet so Drew paints terrains and Godot auto-selects the blended edge tiles.
3. **This rides the AI-gen pipeline Sponsor already approves** (the same path that produces the good characters/mobs), instead of inventing per-surface procedural art that keeps missing.

**Why `create_topdown_tileset` is the RIGHT tool here (the load-bearing engineering fact).** Per `.claude/docs/pixellab-pipeline.md` §"`create_topdown_tileset` is a terrain-TRANSITION tool": this tool is a Wang/corner generator whose *entire purpose* is the transition between two terrains (it was the WRONG tool for a single uniform crafted floor — which is why it was eliminated before). **We now WANT exactly what it makes: soft blended terrain edges.** The doc confirms (addendum 2026-06-07): used *correctly* (deliberate `lower` ≠ `upper`), it **DOES tile seam-free** — the Wang corner-matching works. Its known limits, which this spec designs AROUND, not against:
- **32px hard cap** on `tile_size` (16 or 32). Factored into scale + compositing (§4).
- **Green/mossy skew** at 32px even with max detail. Handled by a doctrine-lock cleanup pass (§5).

## 2. THE TERRAIN SET — terrains + transition pairs

### 2.1 The three terrains

| # | Terrain | Role | Proportion (carried from v2) | Palette (warm S1 ramp, §6) | Baked green? |
|---|---|---|---|---|---|
| **T0** | **DIRT — worn bare earth** | The MAJORITY background ground; the connective tissue between landmarks | **~55–65%** | `#6B5A41` dirt / `#54452F` dirt_deep + worn-lighter/damp-darker | **NO** |
| **T1** | **GRASS — reclaimed patches** | Where feet didn't fall, nature took back: edges, corners, shadowed building bases, around springs/garden | **~20–30%** | `#5C7044` moss/grass mid + a darker `#3E4D2E` shadow + lit `#7E9456` (all sub-1.0) | grass IS this terrain |
| **T2** | **fine-COBBLE — laid paving + the LANE** | The walked lane (the wayfinding spine) + a few surviving cobble patches near landmarks | **~10–20%** | `#7E7460` path base / `#988C76` lit / `#5E564A` shadow / `#3D372F` joint (warm fine-cobble, §6) | **NO** |

This is the v2 three-material composition (`s1-ground-composition-v2.md` §3) — UNCHANGED in intent. What changes: instead of three separately-generated procedural surfaces painted with feathered borders by hand, the three are **Godot terrains** and the boundaries are **AI-generated Wang transition tiles** that Godot autotile places automatically.

### 2.2 The transition pairs (which boundaries need a Wang tileset)

A Godot TerrainSet resolves boundaries between terrains using corner/peering-bit tiles. We generate one **`create_topdown_tileset` Wang set per ADJACENT terrain pair** that actually meets in the yard:

| Pair | Wang tileset | Where it's needed | Reference for the edge feel |
|---|---|---|---|
| **DIRT ↔ GRASS** | **Wang set A** (REQUIRED — the most-used boundary) | Grass reclaiming the dirt at every edge/corner | Stardew `11h19_36` (soft feathered dirt↔grass) |
| **DIRT ↔ fine-COBBLE** | **Wang set B** (REQUIRED — the lane edges) | The fine-cobble lane + cobble patches meeting the dirt field; the lane's whole length | Graveyard Keeper `11h18_12` (worn dirt↔stone), village street `08h00_28` |
| **GRASS ↔ fine-COBBLE** | **Wang set C** (SECONDARY — generate if grass abuts cobble directly) | Where grass creeps onto the cobble lane/patch edge (mossy stone) | `11h55_58` (cobble-into-grass mossy blend), `07h53_24` |

**Connected-tileset chaining (the consistency lever — USE THIS).** `create_topdown_tileset` supports CONNECTED tilesets via **shared base-tile-IDs**: generate set A first (dirt + grass), then generate sets B and C *referencing the same DIRT and GRASS base tiles* so the dirt in A, B matches and the grass in A, C matches across transitions. This guarantees the dirt under a grass blend reads identical to the dirt under a cobble blend — no two-tone-dirt artifact at a three-way junction. **The base tiles (all-corners-same interior) become the three terrain CENTER tiles**; the corner-variant tiles become the blend edges. (Gen plan §4 sequences this.)

Pairs we do NOT generate: there is no GRASS↔GRASS or DIRT↔DIRT "transition" (those are the center tiles, produced as the all-same-corner tile of each set). A full 3-terrain Godot TerrainSet can also be expressed as ONE terrain with three terrain-indices if the chained base-IDs line up — Drew's wiring call (§6); the GEN is per-pair regardless.

**Step 2 — Wang set B: DIRT ↔ fine-COBBLE** (CONNECTED — reuse the set-A DIRT base-tile-IDs)

- `upper_description`: *"fine small warm grey-tan cobblestones, many small set walked-smooth stones, WARM tan not grey, NO moss, NO green"* + `detail`/`shading` = highly detailed (§3 route 1)

**Step 3 — Wang set C: GRASS ↔ fine-COBBLE** (CONNECTED — reuse set-A GRASS + set-B COBBLE base-tile-IDs) — SECONDARY, generate if grass abuts cobble
- `lower_description`: cobble (SAME as set B)
- `upper_description`: grass (SAME as set A)

- `transition_description`: *"grass and moss creeping up between the outer cobblestones, soft green invading the joints at the edge only"*

**One-line gen-plan summary:** three CHAINED `create_topdown_tileset` Wang sets at `tile_size=32`, high top-down — A: dirt↔grass (seeds shared base-IDs), B: dirt↔cobble (connected, reuses dirt), C: grass↔cobble (connected, reuses grass+cobble) — each with a soft `transition_description` + warm-S1 palette guidance, then §5 doctrine-lock to strip the green skew + pin the warm ramp.

## 5. DOCTRINE-LOCK CLEANUP — strip the green skew, pin the warm S1 ramp

The 32px Wang output **skews green/mossy** (documented `pixellab-pipeline.md` addendum). The DIRT and COBBLE terrains must read warm with ZERO unwanted green; only the GRASS terrain (and the grass side of its blends) keeps green.

**Cleanup pass (orch-run, after gen, before handoff):**
1. **Per-tile palette inspection** — pull the palette of each generated tile. Any green-skew on a dirt or cobble tile (or the dirt/cobble SIDE of a transition tile) is wrong.
2. **Region-masked recolor (the right tool):** use **Strategy 5 — single-region hue-mask recolor with luminance preservation** (`pixellab-pipeline.md` §Strategy 5). The green-skew on dirt/cobble fires the moss mask (`g > b + 20`); recolor those pixels to the warm dirt/cobble ramp while PRESERVING luminance (keep the stone/dirt light-dark structure, shift hue warm). This is exactly the validated S1 water-head recolor pattern, applied to strip green from a warm surface. The GRASS terrain's legitimate green (and the grass side of A/C transitions) is left untouched — mask spatially / by tile if needed so a grass blend's green survives.
3. **Doctrine pin (gentle, not a hard lock):** nudge dirt → `#6B5A41`/`#54452F`, cobble → the `S1_PATH_COBBLE_DOCTRINE` warm ramp (§6), grass → `#5C7044` family. Per `pixellab-pipeline.md` §"AI-generated tiles CAN make a continuous floor": Sponsor's feel-approval of the AI look overrides a hard doctrine-lock where they conflict — do a gentle warm nudge that keeps the AI stone/grass structure, not a flat positional set_palette that would collapse it.
4. **All channels sub-1.0** (HDR-clamp-safe, §6).

**Flag to Drew + Sponsor:** if the Wang cobble center tile reads worse than the loved procedural cobble even after cleanup, take §3 route 2 (procedural cobble as center tile, Wang for edges only) — and run BOTH through this same §5 cleanup so the lane is one coherent material.

## 8. GODOT AUTOTILE WIRING GUIDANCE (for Drew — wiring + placement is Drew's, this is enough to wire it)

> Wiring + painting + placement are **OOS for this spec** (Drew owns them). This section gives Drew enough to wire the AI-gen Wang output into a Godot TerrainSet.

**Engine integration shape (confirmed from `S1YardChunk.gd`):** the yard paints into a **`FloorTiles` TileMapLayer** via `set_cell` in `_ready`, using atlas-source-ids in `s1_cloister_yard.tres`, on a **32px game-tile grid**, `grid_w=40 grid_h=24` (1280×768). The terrain set slots directly into this.

**Recommended wiring:**
1. **Import the doctrine-locked Wang PNGs** (one per pair: A dirt↔grass, B dirt↔cobble, C grass↔cobble) as atlas sources in `s1_cloister_yard.tres`. Each 32px tile = one atlas tile (1:1 to the game grid).
2. **Define ONE Godot TerrainSet** on the TileSet with **mode = Corners (and Sides)** — the Wang corner model maps to Godot's corner-peering-bit autotiler. Add three terrain indices: 0=DIRT, 1=GRASS, 2=COBBLE.
3. **Assign peering bits per tile:** for each Wang tile, set the four corner peering-bits to the terrain present at that corner (e.g. an all-dirt corner tile = DIRT on all 4 corners = the DIRT center tile; a dirt-with-grass-NE tile = DIRT on three corners + GRASS on the NE corner). The connected-tileset chaining (§4) means the dirt corner in set A and set B share the same base tile, so the autotiler reads them as the same terrain — no two-tone seam at a junction.
4. **Center tiles:** the all-same-corner tile of each Wang set is that terrain's CENTER (painted in the terrain's interior). If §3 route 2 is taken, swap the COBBLE center tile for the procedural-cobble-salvage tile (same 32px, palette-matched via §5) and keep the Wang cobble tiles only as the dirt/grass-side edge tiles.
5. **Painting (placement):** paint terrains, not tiles — Godot's `set_cells_terrain_connect` (or the editor terrain brush) auto-selects the blend tile at every boundary. Compose per the v2 composition map (§9 cross-ref): DIRT as the broad background field; the fine-COBBLE LANE as a humble worn footpath threading spawn → spine → descent (N/S branch at the central building, well approach); GRASS clustered at corners/edges/shadowed building bases/around springs+garden; surviving COBBLE patches at the well apron + a building entrance + a courtyard remnant.
6. **Z / layers:** all ground terrains floor-layer z=0; boundaries are resolved by the autotiler (one terrain per cell + the blend tiles), NOT by stacking z — avoids the camera-scroll z-fight class (`camera-scroll.md`). No negative z-index (PR #137).
7. **Keep `set_cell`/terrain painting in `_ready`** (deterministic, diff-readable, GUT-testable) per the existing `S1YardChunk.gd` rationale — don't hand-edit binary TileMap data in the .tscn.

**What Drew validates in-context (§7):** every boundary is a soft blend (no hard seam), cobble is fine + warm, dirt is the majority, grass is composed reclamation (not a lawn, not baked speckle), all PL-GND pins hold at game zoom + `char_scale=0.6`.
