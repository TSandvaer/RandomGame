## 1. Build verification + footer probe

The release CI workflow (`release-github.yml:87-94`) DOES stamp `build_info.txt` with `${GITHUB_SHA:0:7}` before the HTML5 export step. The `BuildInfo` autoload (`scripts/debug/BuildInfo.gd:43-58`) reads `res://build_info.txt` first, falling back to `GITHUB_SHA` env, falling back to `"dev-local"`. The HUD footer (`scenes/Main.gd:493-505`) reads `BuildInfo.display_label`.

**The bug:** `export_presets.cfg` sets `export_filter="all_resources"` for every preset — only Godot-recognized resources (`.gd` / `.tscn` / `.tres` / imported assets) get packed into the `.pck`. A plain text file at the project root is NOT a resource, so `build_info.txt` is **excluded from the HTML5 export** even though CI stamped it on the runner filesystem. At runtime the autoload's `FileAccess.file_exists("res://build_info.txt")` returns false (the file isn't in the .pck virtual filesystem), so it falls through to the `dev-local` literal.

This is filed as BB-1 below. The fix is one-line: add `build_info.txt` to `include_filter` for every preset.

## 3. P0 / P1 bug detail

### BB-1 — build_info.txt excluded from HTML5 export

- **Severity:** blocker
- **Area:** build / CI
- **Symptom:** Every HTML5 RC's HUD footer reads `build: dev-local` instead of the actual short SHA.
- **Why it's blocker:** `team/tess-qa/m1-test-plan.md` §"Build identification" mandates that every test run record the build SHA. Sponsor cannot triage which RC they're soaking; Tess cannot pin a bug to a specific commit when the footer lies. The ENTIRE point of the footer is auditability — a dev-local literal in production is functionally a regression of the W2 CI hardening.
- **Repro shape:**
  1. Trigger a release: push a `m1-rc*` tag OR run `release-github.yml` workflow_dispatch.
  2. Download the HTML5 zip artifact.
  3. Unzip + open `index.html` in a browser (any).
  4. Observe HUD bottom-left footer.
- **Reproducibility:** always (deterministic — code path is unconditional).
- **Expected:** Footer reads `build: <7-char short SHA>` matching `${GITHUB_SHA:0:7}`.
- **Actual:** Footer reads `build: dev-local`.
- **Root cause:** `export_presets.cfg` sets `export_filter="all_resources"` for HTML5 / Windows / Linux / macOS presets. This filter only packs Godot-imported resources. `build_info.txt` is a plain `.txt`, so the .pck does not contain it. `BuildInfo._resolve_sha()` (`scripts/debug/BuildInfo.gd:46-52`) does `FileAccess.file_exists("res://build_info.txt")` which returns false at runtime (the file is on the CI runner disk during export but not in the .pck). The function falls to step (2) `OS.get_environment("GITHUB_SHA")` — a no-op on HTML5 — then to step (3) `FALLBACK_SHA = "dev-local"`.
- **Suggested fix scope:** S. Add `include_filter="build_info.txt"` to every export preset block in `export_presets.cfg`. (Alternatively, change BuildInfo to look in `user://` and have CI place it there — but `include_filter` is the clean fix.)
- **Verification gate:** TI-8 in `tests/integration/test_html5_invariants.gd` already asserts `BuildInfo.short_sha` non-empty. Add a paired test that asserts `BuildInfo.short_sha != "dev-local"` when `build_info.txt` is present at `res://`. The native CI passes both because CI runs from the source filesystem (no `.pck` indirection) — only the shipped HTML5 export hits the bug. The right gate is a post-export probe on the .pck contents.

### BB-2 — saved Inventory items silently dropped on every load

- **Severity:** blocker
- **Area:** save / load
- **Symptom:** Quit, reload — every item in the player's inventory + every equipped item disappears. Level + V/F/E + room-progress restore correctly; only inventory state is lost.
- **Why it's blocker:** AC6 (quit + relaunch + state restored) is an M1 acceptance criterion. The existing integration test `tests/integration/test_ac6_quit_relaunch.gd` and `tests/integration/test_m1_play_loop.gd` likely pass because they construct `ItemDef` registry shims at test time — but the production `Main.gd` ships with no-op resolvers. A real player loses every drop they collected the moment they reload.
- **Repro shape:**
  1. New game from title.
  2. Walk to room 1, kill a grunt (will require Devon's combat fix).
  3. Pickup drops; auto-collect into Inventory.
  4. Equip the dropped weapon (Tab → click cell).
  5. Close the browser tab (or quit on native).
  6. Reopen the build → inventory is empty, equipped slot is empty, but level + V/F/E are correct.
- **Reproducibility:** always. Code path is unconditional.
- **Expected:** Per `team/devon-dev/save-format.md`, every saved item rebuilds from `to_save_dict()` via `ItemInstance.from_save_dict(entry, item_resolver, affix_resolver)`, where the resolvers map `StringName id → Resource (ItemDef / AffixDef)` against the live registry.
- **Actual:** `scenes/Main.gd:605-608` builds the resolvers as `noop_item: Callable = func(_id: StringName) -> Resource: return null` and `noop_affix: Callable = func(_id: StringName) -> Resource: return null`. `ItemInstance.from_save_dict` (`scripts/loot/ItemInstance.gd:101-104`) sees `item_def == null` and returns null with a `push_warning("ItemInstance.from_save_dict: unknown item id '%s'" % item_id)`. `Inventory.restore_from_save` (`scripts/inventory/Inventory.gd:270-272`) checks `if inst == null: continue` — every saved item is skipped silently.
- **Root cause:** Devon's PR #107 inline-comment at line 605 says "no-op resolvers in M1; ItemDef registry is stub." This is wrong — `Drew`'s loot system already authors `MobLootSpawner` with real `ItemDef` references; the registry isn't a stub, it's just not wired here. Either (a) `Main.gd` needs to look up `ItemDef`s by id from a real registry node (M1 doesn't have one — needs creation), OR (b) every item def needs a stable `res://` path that `from_save_dict` can `load()` directly.
- **Suggested fix scope:** S/M depending on approach. The minimum viable fix: add an `ItemRegistry` autoload that scans `res://content/items/*.tres` on _ready and builds a `Dictionary[StringName → ItemDef]`, plus the same for affixes; `Main.gd` resolvers call into it. Alternatively (S): the `to_save_dict` already records the def's `res://` path — `from_save_dict` could `load()` directly. Devon's call.
- **Verification gate:** Add an integration test that saves a 2-item inventory + 1-equipped item, reloads via `Main._load_save_or_defaults()`, asserts both stash items + equipped item are restored. The existing `test_m1_play_loop.gd::test_ac6_quit_relaunch_restores_state` (per its docstring) tests this surface — needs to be checked that it actually exercises the production resolver path, not a test-shim.

### BB-3 — no room boundary collision; player walks off into the void

- **Severity:** major
- **Area:** levels / collision
- **Symptom:** Walking past the room's edge takes the player off-screen with no wall stop. Every M1 room (Room01..Room08, BossRoom) is 15×8 tiles = 480×256 px and ships as a `Node2D` with no perimeter `StaticBody2D` walls.
- **Why it's major (not blocker):** The game is technically playable — the camera/viewport is at output resolution and the player can wander back; mobs follow via aggro. But the spec's "stratum room as bounded combat arena" promise from `team/uma-ux/visual-direction.md` and `team/drew-dev/level-chunks.md` is broken in the shipped product. A first-time player who walks down/right past the room boundary sees their character drift into empty world space — a cosmetic-shading-into-functional bug that erodes the "you're in a place" feel.
- **Repro shape:**
  1. New game.
  2. Hold S (move down) for ~2 seconds without engaging the grunt.
  3. Player drifts off the bottom edge of the visible room. (Same on every direction; no walls.)
- **Reproducibility:** always.
- **Expected:** Player collides with a wall at the room's perimeter (or at minimum at the chunk's `bounds_px` rectangle). `RoomGate` is supposed to lock the player in for clear-the-room rooms (Room02..08); even Room01 should bound the play area.
- **Actual:** `scripts/levels/LevelAssembler.gd:32-75` builds `result.bounds_px = Rect2(Vector2.ZERO, size_px)` but does NOT instantiate any `StaticBody2D` walls. The room's `_build()` adds `_assembly.root` (a bare `Node2D` with mob children) to the scene — no collision geometry. `RoomGate` is an `Area2D` (signal-only, masks layer 2 / player) — it triggers `body_entered` for the lock-on-entry contract but does NOT physically block movement. The player's `CharacterBody2D` (`scenes/player/Player.tscn:8`, mask=1, layer=2) collides with layer-1 ("world") bodies — none exist in the level chunks.
- **Root cause:** Walls were never authored. M1 chunks (`s1_room01.tres` ... `s1_room08.tres`, plus `s1_boss_room.tres`) define `size_tiles` + `tile_size_px` + ports + mob_spawns but no wall_tiles or perimeter_geometry. The assembler has no path to spawn walls. Possibly intentional for M1 placeholder fidelity ("rooms have no tiles / sprites" — see BB-8), but the SHIP-FACING behavior (no walls) is what reaches Sponsor.
- **Suggested fix scope:** M. Either (a) `LevelAssembler.assemble_single` adds 4 perimeter `StaticBody2D` rectangles sized to `bounds_px` on every room (cheap, zero authoring cost), or (b) `LevelChunkDef` gains a `walls: PackedVector2Array` field and authors edit each chunk to include a wall outline. (a) is the right M1 fix; (b) is M2.
- **Verification gate:** Paired test: instantiate `Stratum1Room01`, place player at `Vector2(-50, 100)` (outside left wall), call `_physics_process` once, assert player is forced to a position with `x >= 0`. Also add a "you can't walk past x=480 / y=270" probe.

### BB-4 — StatAllocationPanel cannot be reopened after closing mid-bank

- **Severity:** major
- **Area:** ui / stat-panel
- **Symptom:** Player levels up (panel auto-opens at level 2 per LU-05). Player presses Esc to close (banking the unspent point per `level-up-panel.md` HINT_TEXT "<Esc> close (point banked)"). Player wants to come back later to spend the point — the docstring at `scripts/ui/StatAllocationPanel.gd:14-15` says "the player presses P to open" — but there is NO P-key handler anywhere in the file.
- **Why it's major:** Banked stat points become **unspendable** between level-ups. `Levels.xp_to_next` grows per level, so the player can wait minutes-to-hours for the next auto-open trigger before they can spend their banked point. The "[+1 STAT]" HUD pip displays correctly, but pressing P does nothing — the cue is misleading.
- **Repro shape:**
  1. New game.
  2. Gain XP until first level-up (panel auto-opens).
  3. Press Esc (panel closes, point banked, HUD shows [+1 STAT]).
  4. Press P repeatedly. Panel does NOT reopen.
  5. Open Tab (Inventory). Stats column shows the banked point but no UI to spend it.
- **Reproducibility:** always.
- **Expected:** Per `scripts/ui/StatAllocationPanel.gd:14-15` and the HUD pip docstring at `scenes/Main.gd:506-520`, pressing P opens the panel and the player spends the banked point.
- **Actual:** `_unhandled_input` (`scripts/ui/StatAllocationPanel.gd:222-246`) only processes 1/2/3/Enter/Esc when `_open == true`. There is no `KEY_P` branch and no input check when `_open == false`. The auto-open via `Levels.level_up` (line 102-106) is the ONLY path that reveals the panel.
- **Root cause:** docstring says "the player presses P" but the input handler was never written. The hint text inside the panel itself ("<1/2/3> pick <Enter> confirm <Esc> close (point banked)") is consistent — it documents what works WHILE OPEN. The reopen affordance was forgotten.
- **Suggested fix scope:** S. Add a `KEY_P` (or new `toggle_stats` action in `project.godot`) input branch in `_unhandled_input` that calls `open(false)` when `_open == false` AND `PlayerStats.get_unspent_points() > 0`. The Open API (`open(_auto_opened: bool)`) already accepts the manual signal.
- **Verification gate:** Paired test: instantiate panel, give 1 unspent point, send a `KEY_P` `InputEventKey`, assert `panel.is_open() == true`. Also assert it's a no-op when `unspent_points == 0` (don't open an empty panel).

Missing 2 of 7 controls is a 28% miss rate on the onboarding surface. Compounding: combat is currently broken (Devon's `86c9m36zh`), so even if the player guessed LMB/RMB they'd see no hit.
- **Repro shape:**
  1. New game.
  2. Read the boot banner during the title-screen → first-room transition.
  3. Banner does not mention LMB or RMB.
- **Reproducibility:** always (banner content is a string literal).
- **Expected:** Banner reads (per `project.godot:79-87` action map) something like `WASD to move, Shift to sprint, Space to dodge, LMB to attack, RMB to heavy-attack, Tab for inventory, E to interact`.
- **Actual:** Banner is incomplete. Note: I was unable to grep the literal "WASD to move, Shift to sprint, Space to dodge" string in the source — it's not a static literal in any `.gd` / `.tscn` / `.cfg` file under `scripts/` or `scenes/`. It MAY be authored in a content `.tres` not yet readable to me, or it may be a stale dispatch description and the banner ships in some other form. **Action item for triage:** Devon to confirm where the banner is sourced; if it's in a .tres I missed, file as content edit; if it's in `Player.gd` (Devon's lane), file as bug there.
- **Suggested fix scope:** S. Update the banner string to include attack bindings. If the banner is part of `Player.gd` (suspected — that's the file I cannot read this run), Devon owns the edit.

### BB-6 — stacked panel close-order leaks Engine.time_scale = 0.10

- **Severity:** minor
- **Area:** ui / time-scale
- **Repro:**
  1. Open Inventory (Tab) — `Engine.time_scale` snapshot=1.0, set=0.10.
  2. Trigger level-up (e.g. via debug fast-XP chord) → StatAllocationPanel auto-opens. Its `open()` snapshots `_previous=0.10`, sets `0.10` (already 0.10, no observable change).
  3. Close StatAllocationPanel first (Esc). Restores `Engine.time_scale = 0.10` (its snapshot).
  4. Close Inventory (Tab). Restores `Engine.time_scale = 0.10` (its snapshot).
  5. Both panels now closed; world stays at 10% time forever.
- **Expected:** time_scale = 1.0 after both panels close.
- **Actual:** time_scale leaks at 0.10. (The `_exit_tree` guards added in run-019 cover the "panel freed while open" case — they do NOT cover this nested-close-order path.)
- **Suggested fix scope:** S. Either (a) panels coordinate via a shared "time-slow stack" autoload, or (b) the InventoryPanel's `_unhandled_input` Tab-toggle is gated on `not StatAllocationPanel.is_open()` — opening Inventory while StatPanel is open is refused. (b) is simpler.
- **Why P2:** requires a specific sequence that's unlikely to come up organically (most players close inventory before leveling up). HTML5 tab-blur during level-up could trigger it but is rare.

- **Suggested fix scope:** S. Either (a) move spawn to `Vector2(80, 144)` (the entry-port world position, per `Stratum1Room01.gd:53` derivation), OR (b) move grunt B to a higher tile-y (further from the entry) in `s1_room01.tres`, OR (c) use the chunk's `entry_world_pos` returned by `LevelAssembler.AssemblyResult` (which Devon already computes but doesn't use as the player spawn).
- **Why P2:** game is playable; just feels rough. A polished M1 places the player far enough from the first mob to teach controls.
