## P0 finding — fistless start blocks combat loop

### Symptom

Within first ~30 seconds: combat appeared broken. Mobs effectively don't die. Sponsor: *"the mobs dont die from me hitting them."*

### Console trace evidence (excerpted from full run)

```
[combat-trace] Player.try_attack | FIRED kind=light facing=(0.0,1.0)
[combat-trace] Player.swing_wedge | spawned kind=light lifetime=0.100 tween_valid=true alpha=0.55
[combat-trace] Player.swing_flash | tween_valid=true tint=(1.00,0.85,0.60) duration=0.060
[combat-trace] Player.try_attack | POST damage=1 hitbox=@Area2D@205:<Area2D#63266883093>
[combat-trace] Grunt.take_damage | amount=1 hp=50->49
[combat-trace] Grunt._play_hit_flash | sprite tween_valid=true rest=(0.55,0.18,0.22) target=white
[combat-trace] Hitbox.hit | team=player target=Grunt damage=1
...
[combat-trace] Hitbox.hit | team=enemy target=Player damage=5
```

Sustained for ~70 grunt hits across two grunts, each `damage=1`, while player takes `damage=5` per enemy hit.

### Root cause (NOT a regression)

The trace is consistent with the design as documented in `team/DECISIONS.md` `2026-05-02 — Damage formula constants locked`: *"Fist (no weapon) is **flat 1 damage** with no Edge/heavy scaling."* All combat paths function correctly:

- Player attacks fire correctly
- Damage IS dealt (`Grunt.take_damage`)
- Hit-flash visible (PR #140 fix verified working)
- Heavy attack also reads `damage=1` because the heavy multiplier scales weapon damage, not fist
- Player takes `damage=5` per enemy hit (Grunt's authored damage)

The arithmetic: 2 grunts × 50 HP × 1 dmg/swing = 100 fist hits to clear Room01, while losing the HP trade ~4-5x over.

### Real bug — content/onboarding gap

Per `team/uma-ux/player-journey.md` Beats 4-5, Stratum-1 Room01 was supposed to ship with a non-threatening practice dummy + LMB/RMB tutorial prompts, so the player would never face a real grunt fistless. The shipped Room01 has 2 grunts at 50 HP and no tutorial. Both individually pass headless tests; the integration surface is what fails. Same class as the Main.tscn-stub miss.

This was already diagnosed by Drew run-002 on 2026-05-06 and tracked as **CU 028** in `team/log/clickup-pending.md` (the proper Drew Stage 2b fix).

## Re-soak attempt 1 — 2026-05-08 — REGRESSION found

**Artifact:** `embergrave-html5-3937831` (post-PR-145)
**Result:** P0 — bandaid does NOT take effect at integration surface. Soak paused after ~3 minutes.

### Boot prints — Devon's code RAN

```
[Inventory] autoload ready (capacity=24)
[Inventory] starting iron_sword seeded (id=iron_sword damage=6)
[Inventory] starter iron_sword auto-equipped (weapon slot)
```

### But combat trace shows damage=1

Every `Player.try_attack | POST damage=1` (light AND heavy). Grunt drains at 1 HP/swing = same fistless math as before PR #145.

### Sponsor screenshots confirm

- **Tab → Inventory panel:** "Capacity: 0 / 24" — inventory grid is empty, no iron_sword icon visible
- **Stats panel:** "Damage --" / "Defense --" / "Crit --" — no equipped weapon visible to stats UI either
- HP screenshots: 15/100 then 35/100 (player took damage while debugging — also hints at HP regen working between encounters)

### Root cause — Devon Hypothesis 4 confirmed

`Main._ready()` was calling `equip_starter_weapon_if_needed()` BEFORE `_load_save_or_defaults()`. Save-restore step unconditionally resets every equipped slot regardless of save content — silently clobbered the iron_sword three lines after the boot print fired. Both `Inventory._equipped["weapon"]` AND `Player._equipped_weapon` got nulled.

### Why PR #145 tests didn't catch it

AC tests used a stub `Node` instead of a real `Player`. `stub.has_method("equip_item")` returned false; `_apply_equip_to_player` silently skipped the Player-side wiring path. **Same product-vs-component miss class** as the original M1 RC fistless P0 — second instance in a row. Tess's 14 tests asserted Inventory state in isolation; the integration through Main._ready boot order + save-restore was not exercised.

### Fix in flight — PR #146

- `scenes/Main.gd` — move `equip_starter_weapon_if_needed()` to AFTER `_load_save_or_defaults()`
- `scripts/player/Player.gd` — call removed from `Player._ready()`; explanatory comment added
- `tests/test_starting_inventory.gd` — AC(d) tests replace stub with real `Player`, add `Player.get_equipped_weapon() != null` assertion
- `tests/integration/test_starter_weapon_damage_integration.gd` (NEW) — boots Main.tscn, drives `try_attack`, asserts `hitbox.damage == 6` and `grunt.hp == 44` (not 49)

PR #146 awaiting Tess sign-off + integration-test sharpness review (Tess to confirm the new integration test fails pre-fix on the actual Player.try_attack damage path, not just on Inventory state).
